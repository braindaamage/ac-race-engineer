# Implementation Plan: Phase 7.4 — Lap Analysis View

**Branch**: `018-lap-analysis-view` | **Date**: 2026-03-06 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/018-lap-analysis-view/spec.md`

## Summary

Build the Lap Analysis view — a read-only telemetry exploration interface in the desktop app. The user selects a session and sees all laps with summary metrics, selects up to 2 laps to view telemetry traces (throttle, brake, steering, speed, gear) plotted against track position using synchronized Recharts LineCharts, and reviews per-corner data (entry/apex/exit speeds, understeer/oversteer).

Requires one new backend endpoint (`GET /laps/{n}/telemetry`) to serve raw trace data from the existing Parquet cache, plus two additive field changes to existing response models. The frontend replaces the existing AnalysisView placeholder with the full implementation using TanStack Query hooks, Recharts charts, and the existing design system components.

## Technical Context

**Language/Version**: TypeScript (strict mode) + Python 3.11
**Primary Dependencies**: React 18, TanStack Query v5, Recharts, Zustand v5 (frontend); FastAPI, pandas (backend)
**Storage**: Parquet (telemetry cache, read-only), SQLite (session state), analyzed.json (metrics cache)
**Testing**: Vitest + React Testing Library (frontend), pytest (backend)
**Target Platform**: Windows desktop (Tauri v2)
**Project Type**: Desktop app (frontend view + minimal backend extension)
**Performance Goals**: Lap list renders <2s, trace charts render <1s, smooth crosshair interaction at 60fps
**Constraints**: Max 2 laps selected simultaneously, ~500 samples per trace (downsampled), all colors from design tokens
**Scale/Scope**: Sessions with 1-100+ laps, tracks with 0-30+ corners, 5 telemetry channels per chart
