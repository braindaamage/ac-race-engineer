# Implementation Plan: Fix Setup Value Domain Conversion

**Branch**: `034-fix-setup-value-domains` | **Date**: 2026-03-11 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `specs/034-fix-setup-value-domains/spec.md`

## Summary

Fix the setup parameter value pipeline that currently mixes storage indices and physical units, causing the LLM to propose nonsensical values. A new pure-function conversion module translates between AC's three storage conventions (INDEX, DIRECT, SCALED) and physical units at two pipeline boundaries: inbound (summarizer) and outbound (setup writer).

## Technical Context

**Language/Version**: Python 3.11+ (conda env `ac-race-engineer`)
**Primary Dependencies**: Pydantic v2 (models), configparser (ini parsing). No new dependencies.
**Storage**: SQLite parameter_cache table (JSON blobs of ParameterRange). No schema migration needed.
**Testing**: pytest. New unit tests for conversion functions. All 1410+ existing tests must pass.
**Target Platform**: Windows desktop (Tauri app backend)
**Project Type**: Desktop app backend (Python library + FastAPI)
**Performance Goals**: N/A — conversion is O(1) per parameter, negligible overhead
**Constraints**: Pure functions only in conversion module. No LLM involvement in conversion.
**Scale/Scope**: ~8 files modified/created, ~200-300 lines of new code + tests

## Implementation Phases

### Phase A: Conversion Foundation (no existing code changes)

**Goal**: Build and fully test the conversion module before touching any existing code.

**Files**:
- NEW: `backend/ac_engineer/engineer/conversion.py`
- NEW: `backend/tests/engineer/test_conversion.py`

**Details**:

1. **conversion.py** — Pure functions, imports only from models.py:
   - `SCALE_FACTORS: dict[str, float]` — `{"CAMBER": 0.1}` (section prefix → factor)
   - `classify_parameter(section: str, show_clicks: int | None) -> str` — Returns "index", "direct", or "scaled" per FR-001 decision tree
   - `to_physical(storage_value: float, param_range: ParameterRange) -> float` — Converts storage → physical based on `param_range.storage_convention`
   - `to_storage(physical_value: float, param_range: ParameterRange) -> float` — Converts physical → storage with index snapping and clamping
   - `_get_scale_factor(section: str) -> float | None` — Internal helper: checks if section starts with any SCALE_FACTORS key

2. **test_conversion.py** — Comprehensive unit tests:
   - `test_classify_index` — SHOW_CLICKS=2 → "index"
   - `test_classify_direct` — SHOW_CLICKS=0, non-CAMBER → "direct"
   - `test_classify_scaled` — SHOW_CLICKS=0, CAMBER_* → "scaled"
   - `test_classify_none` — SHOW_CLICKS=None → "direct" (Tier 3)
   - `test_classify_unknown` — SHOW_CLICKS=1,3 → "direct"
   - `test_to_physical_index` — INDEX: storage=2, MIN=25500, STEP=4500 → 34500
   - `test_to_physical_scaled` — SCALED: storage=-18 → -1.8
   - `test_to_physical_direct` — DIRECT: storage=18 → 18
   - `test_to_physical_none_convention` — None: passthrough
   - `test_to_storage_index` — INDEX: physical=30000 → storage=1
   - `test_to_storage_index_snap` — INDEX: physical=31000 → storage=1 (snapped)
   - `test_to_storage_index_clamp_low` — INDEX: physical=20000 → storage=0
   - `test_to_storage_index_clamp_high` — INDEX: physical=50000 → storage=max_index
   - `test_to_storage_scaled` — SCALED: physical=-1.0 → storage=-10
   - `test_to_storage_direct` — DIRECT: physical=16 → 16
   - `test_roundtrip_index` — Parametric: for each valid index, to_storage(to_physical(idx)) == idx
   - `test_roundtrip_scaled` — Parametric: to_storage(to_physical(val)) == val ± 1e-9
   - `test_roundtrip_direct` — to_storage(to_physical(val)) == val
   - `test_index_step_zero_fallback` — step=0 in INDEX context → treat as DIRECT

**Validation**: All new tests pass. Zero existing tests touched.

### Phase B: Model & Resolver Changes

**Goal**: Add show_clicks to ParameterRange, read it in the resolver, detect stale caches.

**Files**:
- MODIFIED: `backend/ac_engineer/engineer/models.py`
- MODIFIED: `backend/ac_engineer/resolver/resolver.py`
- MODIFIED: `backend/ac_engineer/resolver/cache.py`

**Details**:

1. **models.py** — Add two optional fields to ParameterRange:
   - `show_clicks: int | None = None` — Raw SHOW_CLICKS value from car setup.ini
   - `storage_convention: str | None = None` — Computed classification: "index", "direct", "scaled", or None
   - Both default to None for backward compatibility
   - Add Field description to SetupChange.value_before and value_after noting they are in physical units

2. **resolver.py** — Modify `_parse_setup_ini()`:
   - After reading MIN, MAX, STEP, DEFAULT: also read SHOW_CLICKS (int, optional)
   - Call `classify_parameter(section_name, show_clicks)` to compute storage_convention
   - Pass both fields to ParameterRange constructor
   - No changes to _resolve_tier1, _resolve_tier2, _resolve_tier3 logic (they all call _parse_setup_ini)
   - Tier 3 doesn't call _parse_setup_ini, so show_clicks remains None (correct behavior)

3. **cache.py** — Modify `get_cached_parameters()`:
   - After deserializing, check if any ParameterRange has `show_clicks is None`
   - If so, return None (cache miss) to trigger lazy re-resolution
   - No Tier 3 exception needed: the DB schema has `CHECK(tier IN (1, 2))`, so Tier 3 results are never cached. Any cached entry with `show_clicks is None` is genuinely stale.
   - Implementation: `if any(pr.show_clicks is None for pr in params.values()): return None`

**Validation**: All existing resolver/cache tests pass. New tests verify SHOW_CLICKS reading.

### Phase C: Inbound Conversion (Storage → Physical)

**Goal**: Convert raw .ini VALUES to physical units before they reach the LLM.

**Files**:
- MODIFIED: `backend/ac_engineer/engineer/summarizer.py`
- MODIFIED: `backend/ac_engineer/engineer/agents.py`

**Details**:

1. **summarizer.py** — Modify `summarize_session()` (NOT `_parse_setup_ini()`):
   - `_parse_setup_ini()` (line 353) is a pure .ini parser that returns `dict[str, dict[str, float | str]]`. It has no access to parameter_ranges and should remain unchanged.
   - The conversion happens AFTER `_parse_setup_ini()` returns, in `summarize_session()` at line ~79-80:
     ```
     active_setup_parameters = _parse_setup_ini(setup_ini_contents)
     # NEW: convert raw storage values to physical units
     if active_setup_parameters and parameter_ranges:
         for section, params in active_setup_parameters.items():
             pr = parameter_ranges.get(section)
             if pr and isinstance(params.get("VALUE"), (int, float)):
                 params["VALUE"] = to_physical(params["VALUE"], pr)
     ```
   - Add `parameter_ranges: dict[str, ParameterRange] | None = None` keyword parameter to `summarize_session()`
   - If parameter_ranges is None (backward compatibility), skip conversion (DIRECT passthrough)

2. **agents.py** — Update `analyze_with_engineer()` and `_build_user_prompt()`:
   - `analyze_with_engineer()` already receives `parameter_ranges`. Pass them to `summarize_session()`.
   - `_build_user_prompt()` reads `active_setup_parameters` which are now in physical units — no changes needed to prompt building.

**Validation**: User prompt now shows physical values. Test with INDEX parameter: VALUE=2 → shown as 34500.

### Phase D: Outbound Conversion (Physical → Storage)

**Goal**: Convert LLM-proposed physical values back to storage format before writing to .ini.

**Files**:
- MODIFIED: `backend/ac_engineer/engineer/setup_writer.py`

**Details**:

1. **setup_writer.py** — Modify `apply_changes()`:
   - Add `parameter_ranges: dict[str, ParameterRange] | None = None` parameter
   - Before writing each VALUE to the .ini file, call `to_storage(effective_value, param_range)` if parameter_ranges is provided and the section exists in ranges
   - If parameter_ranges is None or section not found, write value unchanged (backward compatibility)
   - The effective_value at this point is already validated/clamped in physical space by `validate_changes()`

2. **agents.py** — Update `apply_recommendation()`:
   - Pass `parameter_ranges` to `apply_changes()` so it can perform the conversion

**Validation**: Test with INDEX parameter: LLM proposes 30000 → written as VALUE=1.

### Phase E: Pipeline Integration & value_before/value_after

**Goal**: Ensure end-to-end flow works and display values are physical.

**Files**:
- MODIFIED: `backend/ac_engineer/engineer/agents.py`
- MODIFIED: `backend/ac_engineer/engineer/__init__.py`

**Details**:

1. **agents.py** — Verify value_before and value_after are physical:
   - `value_before` comes from `active_setup_parameters` (now physical after Phase C) — correct
   - `value_after` is the LLM's proposed physical value — correct
   - `_post_validate_changes()` may clamp value_after but stays in physical space — correct
   - No code changes needed if Phases C and D are correct, but verify with integration test

2. **__init__.py** — Export conversion functions:
   - Add: `classify_parameter`, `to_physical`, `to_storage` to public exports
   - Add: `SCALE_FACTORS` constant for visibility

**Validation**: End-to-end test with FunctionModel: ARB_FRONT INDEX → physical in prompt → physical proposal → storage in .ini.

## Risk Analysis

| Risk | Impact | Mitigation |
|------|--------|------------|
| Existing tests break due to ParameterRange field additions | Medium | New fields default to None. Pydantic v2 ignores extra fields in model_validate. |
| Cache deserialization fails with old entries | Medium | Pydantic v2 handles unknown fields gracefully. Staleness check returns None → re-resolve. |
| Summarizer called without parameter_ranges | Low | New parameter defaults to None → no conversion (DIRECT passthrough). |
| apply_changes called without parameter_ranges | Low | New parameter defaults to None → no conversion. Existing callers unaffected. |
| Mod car with unexpected SHOW_CLICKS values | Low | Unknown values default to DIRECT (no conversion). Safe fallback. |
| Non-CAMBER scaled parameter discovered in future | Low | Add entry to SCALE_FACTORS dict. Single-line change. |
