# Implementation Plan: Session Discovery

**Branch**: `011-session-discovery` | **Date**: 2026-03-05 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/011-session-discovery/spec.md`

## Summary

Session Discovery (Phase 6.2) connects the FastAPI backend to the telemetry files generated by the AC in-game app. A `watchdog`-based file watcher monitors `~/Documents/ac-race-engineer/sessions/` for new CSV + meta.json pairs and automatically registers them in SQLite. Four REST endpoints expose session listing, detail, manual sync, and deletion. The existing `sessions` table is extended with additive columns (state, session_type, csv_path, meta_path) via idempotent migrations. The scanner logic is pure Python (no HTTP coupling), shared between the watcher and the sync endpoint.

## Technical Context

**Language/Version**: Python 3.11+ (conda env `ac-race-engineer`)
**Primary Dependencies**: FastAPI, watchdog>=4.0, Pydantic v2
**Storage**: SQLite (stdlib sqlite3) — existing `data/ac_engineer.db`
**Testing**: pytest with `tmp_path` fixtures, httpx `TestClient` for endpoint tests
**Target Platform**: Windows 10/11 (primary), cross-platform compatible
**Project Type**: Desktop app backend (API server)
**Performance Goals**: Session list <1s for 500 sessions, file detection <5s after write completes
**Constraints**: Single-user, local-only, no concurrent server instances
**Scale/Scope**: Hundreds of sessions per user, single sessions directory

**Post-design re-check**: All gates pass. The watcher module (`api/watcher/`) contains OS-level filesystem monitoring which is inherently infrastructure, but the session registration logic (scanning + metadata extraction) is pure and testable without a running server.
