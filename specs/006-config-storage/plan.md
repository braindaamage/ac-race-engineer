# Implementation Plan: Config + Storage (Phase 5.1)

**Branch**: `006-config-storage` | **Date**: 2026-03-04 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/006-config-storage/spec.md`

## Summary

Build two new `ac_engineer` submodules — `config/` and `storage/` — that provide the persistent infrastructure layer for AC Race Engineer. The config module manages user settings (AC install path, setups path, LLM provider/model) via a Pydantic v2 model serialized to JSON with atomic writes. The storage module provides SQLite-based CRUD for sessions, recommendations (with setup_changes), and conversation messages using stdlib sqlite3 only. Both modules follow the existing codebase patterns: pure functions, `__all__` exports, `**overrides` test builders, ~40 pytest tests.

## Technical Context

**Language/Version**: Python 3.11+ (conda env `ac-race-engineer`)
**Primary Dependencies**: Pydantic v2 (existing), stdlib only for new code (sqlite3, uuid, pathlib, json, logging)
**Storage**: JSON file (`data/config.json`) for config; SQLite (`data/ac_engineer.db`) for relational data
**Testing**: pytest with `tmp_path` fixture — never touch `data/`
**Target Platform**: Windows desktop (Assetto Corsa runs only on Windows)
**Project Type**: Backend library module (consumed by future API + CLI layers)
**Performance Goals**: All operations < 100ms for up to 1,000 records
**Constraints**: No new external dependencies; atomic config writes (tmp + rename); `read_config()` never raises
**Scale/Scope**: Single-user desktop app; 4 SQLite tables; ~8 public functions
